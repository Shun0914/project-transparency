# 課題定義

## 解決したい課題の明確化

### 困りごとのリストアップ
- 引き継ぎ時に「誰に何を聞けばいいか」が暗黙知になっていて、ドキュメントに書かれていない
- 複数の業務それぞれに対して、連絡先・判断基準・トラブル対応をゼロから手作業でまとめる必要がある
- マニュアルはあっても、実務で必要な「困った時の人脈」や「判断パターン」が可視化されていない
- 引き継ぎのタスク洗い出し自体が属人的で、抜け漏れが発生しやすい
- **最も本質的な問題：「何が書かれていないか」が誰にも分からない**

### 誰が、どんな状況で、なぜ困っているか

**誰が：** 退職・異動・休職する社員と、その引き継ぎを受ける後任者、そしてプロジェクト全体を管理するマネージャー

**どんな状況で：** 
- 退職・異動の1〜2ヶ月前に引き継ぎ準備を始める段階
- 引き継ぎドキュメントは一応あるが、「誰に聞けばいいか」「どう判断すればいいか」などの情報が膨大に抜けている
- 後任者が決まってから実際に引き継ぐまでの期間が短い（1ヶ月程度）
- 複数の業務を並行して引き継ぐ必要がある
- **プロジェクトメンバー全員が「全体像」をどれだけ把握しているか分からない**

**なぜ困っているか：**

1. **情報の分散問題**
   - 社内の「誰が何を知っているか」が可視化されておらず、困った時にどこに聞けばいいかが分からない

2. **ドキュメント化されない情報の問題**
   - 「実際にはこうやってる」という情報を、手作業でゼロからドキュメント化する必要があり、時間がかかりすぎる
   - スプレッドシートもConfluenceもあるのに、誰も書かない

3. **引き継ぎ漏れのリスク**
   - タスクの洗い出し自体が属人的で、「何を引き継ぐべきか」の棚卸しに抜け漏れが発生する
   - 「何が足りていないか」が可視化されていない

4. **非効率なナレッジ探索**
   - 後任者が実務を始めてから「あれ、これどうするんだっけ？」となった時、過去のSlack/Teams/メールを手動で探す必要がある

5. **対話のきっかけがない**
   - プロジェクト全体で「何が分かっていて、何が分かっていないか」を議論する機会がない
   - 定量的な指標がないため、「ちゃんと引き継げてる」の基準が曖昧

## なぜ既存ツールでは解決しないのか

### Confluence / Notionの限界
- ドキュメントを書く「場所」は提供するが、「書く動機」は提供しない
- 結果：誰も書かない、更新されない、古い情報が残る

### なぜ誰も書かないのか？
1. **面倒くさいから**
   - 業務で忙しいのに、ドキュメント書く時間なんてない
   - 「後でやろう」→永遠にやらない

2. **インセンティブがないから**
   - 書いても自分にメリットがない（むしろ時間の無駄）
   - 評価されない、誰も見ない

3. **「退職時に困る」が遠すぎるから**
   - 今は辞める予定ないし、関係ない
   - 切迫感がない

4. **何を書けばいいか分からないから**
   - 「引き継ぎ事項を書いてください」って言われても、何書けばいいの？
   - 自分が当たり前にやってることは、暗黙知すぎて言語化できない

### 「暗黙知」という言葉の罠

「暗黙知を可視化する」という表現は便利だが、実は曖昧すぎる。

本当の問題は：
- **「ドキュメント化されていない情報」** - 言語化できないわけではなく、単に書かれていないだけ
- **「散らばっている情報」** - Slack、メール、個人の頭の中に分散している
- **「何が書かれていないかが分からない」** - これが最も本質的

## 市場規模とビジネスの可能性

### 市場規模
- 日本だけでも年間数百万件の退職・異動が発生
- 企業の引き継ぎコストは莫大（属人化による業務停滞、引き継ぎ期間の人件費、ナレッジロスによる機会損失）
- SaaS型で企業向け課金モデルなら、エンタープライズ市場も狙える

### スケーラビリティ
- 業界・職種問わず普遍的な課題
- 一度導入されれば、継続的に使われる（退職・異動は常に発生）
- ネットワーク効果：使えば使うほど組織のナレッジが蓄積される

### 競合優位性
- 既存ツール（Confluence、Notion等）は静的なドキュメント管理に留まる
- 「ドキュメント化率のスコアリング」という切り口は未開拓
- 組織の健全性を測る「体温計」としての機能は他にない

### 収益性
- 企業向けサブスクリプションモデル（従業員数課金 or プロジェクト数課金）
- 大企業ほど引き継ぎコストが高く、導入インセンティブが大きい
- HR Tech / Productivity Toolsとして、既存の企業ITスタックに組み込みやすい

## なぜ今、このソリューションが存在しないのか？

### 1. タブーだから
- 「退職」を前提にしたツールは、企業文化的にNG
- 人事部が「辞める準備を促進するツール入れましょう！」とは言えない
- 誰も正面から取り組めなかった

### 2. インセンティブ設計が難しいから
- 「将来辞める時のために今やる」は動機が弱い
- 切迫感がないと人は動かない

### 3. 誰が買うのか不明確だったから
- 社員個人は払わない
- 会社は「辞める準備させるツール」に投資しない
- ビジネスモデルが見えなかった

### 4. 既存ツールで「一応」解決してると思われていたから
- Confluence、Notion、スプシで「書けばいい」
- 問題は「誰も書かない」ことだけど、それは「運用の問題」として片付けられてた

### 5. 技術的なハードルがあった（過去）
- AIで情報を抽出・整理するのは、数年前まで難しかった
- Slack/Teams連携も、最近ようやく一般的に

### なぜ「今」なら可能なのか
- **働き方が変わった**：終身雇用の崩壊、転職が当たり前、「辞める前提」が普通に
- **AIの進化**：自然言語処理で情報抽出が現実的に
- **ツール疲れ**：「また新しいツール？」ではなく、「本質的に解決するツール」が求められている
- **価値観の変化**：「いつでも辞められる自由」が重視される時代
