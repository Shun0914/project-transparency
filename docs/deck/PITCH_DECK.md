# Project Transparency - プロジェクトの透明性を可視化する

---

## 課題：何が書かれていないか、誰も分からない

### 引き継ぎで起こる問題

- 退職・異動時に「誰に何を聞けばいいか」がドキュメントに書かれていない
- マニュアルはあっても、実務で必要な「判断パターン」や「人脈」が可視化されていない
- 引き継ぎのタスク洗い出し自体が属人的で、抜け漏れが発生する

### 最も本質的な問題

**「何が書かれていないか」が誰にも分からない**

---

## なぜ既存ツールでは解決しないのか

### Confluence / Notionの限界

- ドキュメントを書く「場所」は提供するが、「書く動機」は提供しない
- 結果：**誰も書かない、更新されない**

### なぜ誰も書かないのか？

1. **面倒くさい** - 業務で忙しいのに、時間がない
2. **インセンティブがない** - 書いても自分にメリットがない
3. **切迫感がない** - 「退職時に困る」が遠すぎる
4. **何を書けばいいか分からない** - 暗黙知すぎて言語化できない

---

## コアアイデア：%スコアリング

### シンプルな問いかけ

プロジェクトメンバー全員に聞く：

**「このプロジェクト、何%ドキュメント化されてると思う？」**

### 例

- Aさん：「70%」
- Bさん：「60%」
- Cさん：「80%」
- Dさん：「50%」

→ **平均：65%**  
→ **残り35%は何？**

---

## なぜこれが機能するのか

### 1. 認識のズレが可視化される

- 「みんな分かってる」と思ってたのに、実は分かってなかった
- 数字があるから、ごまかせない
- Dさんだけ50%と言っている → 何が見えていないのか？

### 2. 対話のトリガーになる

- 「なんで30%足りないと思うの？」
- 「私は20%と思ったけど、あなたは40%と言ってる。何が違う？」
- **議論が生まれる**

### 3. 主観でいい

- 正解はない
- みんなの主観の集合が、真実に近づく
- 完璧じゃなくていい

---

## 役職による重み付け

### 経験値の違いを考慮

- **PL（Project Leader）**: 重み 3
- **PM（Project Manager）**: 重み 2
- **Member**: 重み 1

### 加重平均の計算

```
加重平均 = Σ(スコア × 重み) / Σ(重み)
```

**例：**
- PL（重み3）: 85点
- PM（重み2）: 75点
- Member（重み1）: 70点

→ 加重平均 = (85×3 + 75×2 + 70×1) / 6 = **78.3点**

---

## これは「引き継ぎツール」ではない

### プロジェクトの全体像を測る体温計

#### 日常的に使う価値

- 週次で「今週、%変わった？」
- 新しいメンバーが入った時「あなたは何%だと思う？」
- プロジェクトが複雑になった時「%下がってない？」

#### 組織の健全性の指標

- チーム全体の平均が70%を下回ったら危険信号
- メンバー間で認識のズレが大きい → コミュニケーション不足
- **組織のバイタルサイン**

---

## 応用シーン

### オンボーディング

- 新入社員「私は今、30%しか理解してない」
- 1ヶ月後「50%になった」
- 3ヶ月後「70%になった」
- **成長が可視化される**

### プロジェクトの引き継ぎ

- 前任者「私は90%理解してる」
- 後任者「私は今、20%」
- 1週間後「40%」
- 1ヶ月後「70%」
- **引き継ぎの進捗が見える**

## 実装：シンプルなMVP

### 技術スタック

- **バックエンド**: FastAPI + SQLite
- **フロントエンド**: Next.js + TypeScript + Tailwind CSS
- **可視化**: Recharts

### 機能

1. **プロジェクト作成** - プロジェクト名とドキュメントURL登録
2. **メンバー登録** - 名前、役職（PL/PM/Member）を登録
3. **スコアリング** - 各メンバーが0-100%でスコア入力
4. **ダッシュボード** - 加重平均、個別スコア、推移グラフを表示

---

## デモ：ダッシュボード画面

### 表示内容

#### 加重平均スコア
- プロジェクト全体の透明性スコア（大きく表示）
- 最終更新日時

#### メンバー別スコア
- 各メンバーの最新スコアとコメント
- 役職と重みが表示される

#### スコア推移グラフ
- 時系列でスコアの変化を可視化
- チームの改善状況が一目で分かる

---

## データベース設計

### シンプルな3テーブル構成

1. **projects** - プロジェクト情報
   - id, name, document_url, created_at

2. **members** - メンバー情報
   - id, project_id, name, role, email, created_at

3. **scores** - スコアリング履歴（全て保存）
   - id, member_id, score, comment, created_at

### 重要な仕様

- スコアは履歴として全て保存（更新ではなく追加）
- プロジェクト削除時、メンバーとスコアも連動削除（CASCADE）

## なぜ今、このソリューションが必要なのか

### 1. 働き方が変わった

- 終身雇用の崩壊
- 転職が当たり前
- 「辞める前提」が普通に

### 2. 属人化のリスクが顕在化

- リモートワークで暗黙知が共有されにくい
- 急な休職・退職に対応できない
- 組織の脆弱性

### 3. 価値観の変化

- 「いつでも辞められる自由」が重視される時代
- 心理的安全性の確保
- 透明性への期待

## ビジョン：いつでも辞められるからこそ、辞めなくなる

### 哲学的な背景

思想家ナシーム・タレブは、退職届を机の引き出しに忍ばせていた。  
それがあるだけで、今日という日が「ご褒美」に感じたという。

### 逆説的な真実

**「いつでも辞められる」と分かると：**

- プレッシャーから解放される
- 今日という日が「義務」から「選択」に変わる
- 心理的な余裕が生まれる
- **結果的に辞める理由が減る**

---

## 組織に起こる変化

### 従業員全員が「いつでも辞められる状態」を持つと

1. **パワーバランスが健全化する**
   - 非対称な権力関係が崩れる
   - 上司と部下が対等な関係になる

2. **ストレスが減り、パフォーマンスが上がる**
   - 「辞められない」プレッシャーがなくなる
   - 心理的安全性が生まれ、創造性が高まる

3. **属人化が減る**
   - みんなが「いつでも引き継げる状態」を維持
   - 情報がオープンになる
